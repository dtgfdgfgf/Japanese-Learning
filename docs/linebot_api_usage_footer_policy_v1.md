# LINE Bot「API 使用度」顯示與模型切換政策（產品規格）

版本：v1.0  
適用範圍：本專案 LINE Bot（聊天即 UI）  
目標：讓使用者在 **不理解配額/tiers/限流** 的情況下，仍能 **一眼判斷用量**、並以 **一鍵操作** 決定「繼續用免費 / 換另一個免費 / 升級付費」。

---

## 1. 核心原則

1) **不要暴露供應商配額**（RPD/RPM/TPM 等）  
- 這些多半是 *per project*、且會隨帳號/使用 tier 變動；對一般使用者是噪音。  
- 我們改為顯示「系統分配給每位使用者的免費額度」，形成可理解、可控的產品體驗。

2) **三檔位對用戶呈現為「模式」而非「模型名稱」**  
- 使用者選擇的是體驗（省錢/推薦/嚴謹），不是供應商與 model id。  
- 後端自行映射：mode → model。

3) **避免指令式設定**  
- LINE 不適合「去設定頁」；預設用 Quick Reply 一鍵切換。  
- 指令僅保留給 power user / debug（可選，且要少）。

4) **每次回覆都附 Usage Footer（2 行為主）**  
- 任何時刻使用者都能看到：今天用到哪、這次用多少、怎麼切換。

---

## 2. 用戶可見的三種模式（Mode）

- **省錢（免費）**：高頻短問答、查詞、快速回覆；輸出偏短。  
- **推薦（免費）**：一般對話/練習/摘要；品質/速度/成本平衡（建議預設）。  
- **嚴謹（付費）**：長輸出、複雜推理、需要更高正確率與穩定性；成本較高。

> 模式標示規則：對用戶永遠顯示「省錢/推薦/嚴謹」與「免費/付費」，可選擇在括號顯示實際模型（通常不建議）。

---

## 3. 「API 使用度」定義（你要顯示的用量指標）

### 3.1 必顯欄位（讓使用者一眼決策）

每次 AI 回覆末尾固定顯示：

1) **今日用量**：`已用 tokens / 每日上限 tokens` + 百分比  
2) **本次用量**：`input tokens + output tokens`（或合計 tokens）  
3) **目前模式**：省錢/推薦/嚴謹（免費/付費）  
4) **一鍵切換入口**：Quick Reply buttons（省錢/推薦/嚴謹/付費模型）

### 3.2 顯示格式（建議 2 行）

**版本 A（精簡、最推薦）**
```
——
📊API使用度：今日 23%（11.5k / 50k tokens）｜本次 420 in + 180 out
⚙️模式：推薦(免費)｜切換：〔省錢〕〔嚴謹〕〔付費模型〕
```

**版本 B（含「本次預估 $」以促進升級決策）**
```
——
📊API使用度：今日剩 38.5k tokens（23% 已用）｜本次 600 tokens
💳若改用付費(嚴謹)：本次約 $0.0012｜切換：〔省錢〕〔推薦〕〔付費模型〕
```

---

## 4. 用量計算策略（產品化關鍵）

### 4.1 「每日免費額度」以系統分配為準

- 每位使用者都有 `daily_cap_tokens_free`（例如 50,000 tokens/day）。  
- 每次請求完成後，將 `prompt_tokens + completion_tokens` 累加到 `daily_used_tokens`。  
- Footer 顯示：
  - `used_pct = daily_used_tokens / daily_cap_tokens_free`
  - `remaining = max(0, cap - used)`

> 這使得「API 使用度」對使用者永遠可理解，也讓你能控成本與公平分配。

### 4.2 Token 來源

- 從各家 API 回傳的 usage 欄位取得（或你在 middleware 內估算/統一）。  
- 最終統一成：
  - `input_tokens`
  - `output_tokens`
  - `total_tokens = input + output`

---

## 5. 模式 → 模型映射（後端）

### 5.1 建議映射（可調整）

- 省錢（免費） → Gemini 2.5 Flash-Lite（或同等免費/低價模型）
- 推薦（免費） → Gemini 2.5 Flash（或你定義的免費穩定檔）
- 嚴謹（付費） → Claude opus 4.5 (較貴) Claude Sonnet 4.5 、 Gemini3 Pro

> 原則：同一模式可設多個候補以做 fallback（同檔位備援）。

### 5.2 fallback 規則（建議）

- 若主模型超時/429/5xx：
  - 立刻切同模式備援模型重試（限制 1 次）。
- 若仍失敗：
  - 回覆「目前服務繁忙」並提示稍後再試（避免無限重試）。

---

## 6. 何時提示使用者升級（避免像推銷）

只有在「明顯需要更強」或「免費快用完」時才提示：

### 6.1 觸發條件（任一成立）

- 使用者連續 2 次出現負向訊號：  
  - 「答錯」「不對」「再仔細」「不夠嚴謹」「請查證」「講得太表面」等
- 任務類型判斷為長輸出/高風險：  
  - 題庫生成、長文改寫、嚴謹解釋、需要引用/查核（依你的分類器）
- 免費額度剩餘 < 15%（或你設定的門檻）

### 6.2 提示文案（加在 Footer 前或第二行）

```
🔎建議：這題需要更嚴謹推理，可切到〔嚴謹(付費)〕提高正確率
```

---

## 7. Quick Reply 設計（一鍵切換）

### 7.1 按鈕集合（建議）

- 〔省錢〕 → set_mode(cheap)
- 〔推薦〕 → set_mode(balanced)
- 〔嚴謹〕 → set_mode(rigorous)
- 〔付費模型〕 → 打開付費選擇（或直接切嚴謹，視產品策略）
- （可選）〔查看用量〕 → 回傳更完整用量卡片（但別每次都顯示）

### 7.2 行為規則

- 使用者點擊後：
  - 立即回覆：「已切換為 XX 模式」+ 下一次對話開始生效
  - 並將偏好寫入 user profile（持久化）

---

## 8. 資料結構（DB schema / user profile）

最小必要欄位：

- `user_id`（LINE userId）
- `mode`：`cheap | balanced | rigorous`
- `daily_cap_tokens_free`：int（預設 50_000）
- `daily_used_tokens`：int（每日累積）
- `reset_at`：datetime（每日重置時間）
- `last_input_tokens`：int（可選，用於顯示本次）
- `last_output_tokens`：int（可選）
- `last_total_tokens`：int（可選）

> 若你有付費方案：另加 `plan`、`billing_status`、`paid_monthly_cap` 等。

---

## 9. 重置與時間策略

- 建議以「使用者所在時區（Asia/Taipei）」做日切，最符合直覺。  
- 每日第一次請求時檢查是否跨日：
  - 若跨日：`daily_used_tokens = 0`，更新 `reset_at`。

---

## 10. 成本估算（可選但很有價值）

如果你要在 Footer 顯示「本次若改付費約 $X」：

- 建立 `pricing_table`（每個模型的 input/output 單價，$/MTok）
- `estimated_cost = (input_tokens/1e6)*price_in + (output_tokens/1e6)*price_out`
- 顯示時四捨五入到 4~6 位小數（避免顯示 0）

> 注意：若你採「嚴謹模式才付費」策略，顯示的是「若切嚴謹，本次約 $X」。

## 12. 非功能性要求（NFR）

- **一致性**：Footer 永遠出現在回覆末尾，避免 UI 忽上忽下。  
- **可讀性**：2 行為主；避免長段落。  
- **可靠性**：即使 usage 取不到，也要降級顯示（例如只顯示模式 + 切換）。  
- **成本控制**：嚴謹模式需設 max_output_tokens 與超額保護（例如達上限自動降回免費）。

---

## 13. 失敗/異常情境處理

1) **取不到 token usage**  
- Footer 顯示：
  - `📊API使用度：暫時無法取得（稍後會恢復）｜本次：—`
  - 保留切換按鈕

2) **429 / 超出速率限制**  
- 先走同模式 fallback  
- 若仍失敗：回覆「服務繁忙，請稍後再試」+ 建議切到另一免費模式（若可降低負載）

3) **免費額度用盡**  
- 自動鎖定免費模式輸出（或縮短回答）
- 清楚提示：「今日免費額度已用完，可明日重置或切付費」

---

## 14. 交付清單（你要實作的最小集合）

- [ ] Middleware：統一收集 input/output tokens → 記錄到 DB  
- [ ] User profile：mode、cap、used、reset 邏輯  
- [ ] Footer renderer：把 usage + 模式 + 按鈕渲染成兩行文字（+ quick reply）  
- [ ] Mode switch handler：處理 quick reply payload → 更新 user.mode  
- [ ] （可選）Cost estimator：模型單價表 + 本次估算顯示  
- [ ] （可選）Fallback router：同模式備援 + 超時重試一次
